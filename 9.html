Welcome to the 9th Lesson

If you viewed the previous lessons on a mobile device you may have noticed that they look like shit. For starters the text appears too small. Zoom in, and the text doesn't wrap, which means you have to scroll left and right. Terrible! This is a common problem and should be solved with good design.

One solution is called Responsive Web Design. Responsive web design is an industry buzzword like User interface plasticity, or Progressive enhancement, or Graceful degradation, or Adaptive web design, or Content fluidity (or is it Fixity?). Whenever people talk about these concepts and how they're different, I want to tell them to shove their Corporate Synergy up their ass, I mean, up their Disrputive Paradigm Shifting Strategic Verticals of Adaptively Responsive Web Design.

Let's avoid the nonsense. We just want our webpage to not look like shit.


Keep it Simple, Stupid

This is the KISS principle. It applies to everything in life. Do NOT make things uncessarily complicated. Add this single line into the <header> element of the HTML.

<meta name="viewport" content="width=device-width, initial-scale=1.0">

The device-width is the width of a device's screen (like a smartphone screen). This will set the webpage to that width. In other words, rotating the device will do what we want it to do. No more side scrolling!

The "initial-scale=1.0" prevents an initial zoom level (so we're not zoomed way out). For a properly designed website, this is all that's needed to get things working well on a smartphone. However, if we've got gigantic fixed-width images and our content doesn't scale to small screens, then we've got a lot more work to do to avoid side-scrolling.


Image-sizing

Images can be sized (relative to their container) by using a %, like so:

<img src="img_kitten_bw_900x240.jpg" style="width:90%">

Resize the browser and make it really small to see how this works. If you view this on smartphone, rotate your display. The image should scale appropriately.


@media queries

CSS allows different styles for different screen sizes.

There are variety of ways to do this, most common is a "media query", like so:

@media only screen and (min-width: 720px) { ... }

Any CSS inside the above media query will apply ONLY to screens that are 720 pixels or wider. Unfortunately, this can get complicated fast. Let's remember the KISS principle. Start simple with a design that looks good on a small display (Mobile First, another buzzword), and then add styles for larger displays.

Here's a simple example:

#body { font-family: Sans-Serif; padding: 10px 10px 3em 20px; max-width: 800px; } #header { border-bottom: 3px solid black; margin-bottom: 10px; padding: 20px; font-size: 20px; background-color: #333; color: #d0d0d9; } @media only screen and (min-width: 720px) { #header { padding: 20px 60px; margin-bottom: 20px; font-size: 32px; } #body { padding: 10px 10px 3em 60px; } }

This is an easy way to make sure our webpage doesn't look like shit. Just remember to Keep it Simple, Stupid.


Make it Complicated

This is the opposite of the KISS principle. For example: Why have 20 lines of effective CSS when we can have 5,000 lines of convoluted nonsense? Obviously, there's no job security in 20-lines of CSS. It should go without saying that this is a terrible idea, but unfortunately there are many unnecessary complications within modern web design. As we'll see, website font-sizing is a great example of this kind of Make it Complicated principle.

CSS Font-sizing

In CSS, we can specify a font-size in "px", "pt", "em", "ex", "%", "mm", "in", "ch", "vw", "rem", and maybe a dozen others. Most of these, like "px" and "pt", behave the same but offer different numeric measurements (like celsius and fahrenheit), and some function as multipliers, like "%" and "em". So really, it's either a measurement or a multiplier, that's it.

You may wonder, why are there so many different ways to specify font size?

As best as I can tell, the W3C is fighting an endless battle between people that went to art school and the people who create the browsers. Most of the font-sizes come from the world of print, legacy traditions of type-setting and typography, a world where we have universal standards of size: an inch is always an inch, a millimeter is always a millimeter, and 25.4 millimeters is 1-inch on every single piece of paper that has ever existed. On a webpage these rules don't apply. If we project our screen onto the side of a building, an inch can become a meter. On a modern smartphone an inch contains more pixels than desktop displays did back when the first CSS standard was written.

I suspect that most designers prefer to use "em" because they went to art school and learned about typography. There are some technical differences between "em" and "px" (pixel), but most designers can't competently explain the difference and prefer "em" out of habit, and in many cases they don't even use it correctly. You'll find misuse of "em" sizing has led to a lot of awkwardly oversized text and cross-device problems. It's sometimes described as the least worse option, but simply leaving out font-size is a much better option (because most devices provide sensible defaults).

For example, here's an extremely common style that should never be used:

html { font-size: 62.5%; }

The 62.5% is so common that it's almost universal. What this does is reduce the browsers default font-size (16px) by 62.5% (to make it 10px). The reason most often cited is that "em" will now match "px", so after the 62.5%, 1.6em is the same as 16px. For similar reasons we may see 0.625em. To see why this is unecessarily complicated, let's look at the following example:

html { font-size: 62.5%; } body { font-size: 1.6em; }

Yep, nothing. 1.6em is equivalent to 160%, 62.5% of 160% is just 100%, and we've accomplished nothing.

But wait, the designers say we should use "em" to have consistent (and resizeable) text across all browsers and devices.

But is that true?

Well, a long time ago, there was a problem in IE6 (a browser that hasn't been used since 2006). In IE6 a font-size in "px" could not be modified by the user (an accessibility problem). IE6 came out in 2001. This is back when people were using <table>'s to design websites. There were no iPhones. This was before 9/11.

Basically, IE6 had terrible support for CSS. IE6 had a lot of problems with "em" as well. One particular fix involved setting the html font-size to 100%, which I guess got people into the habit of using a %, and this led to the 62.5% nonsense (ironically, "em" still doesn't work properly in IE6 even with the 62.5% trick). Think how odd it is that decades later people "reset defaults" (font-size: 100%), and then set font-size to 62.5% in order to use "em" as a numerical equivalent of "px". Instead, they could do nothing and it would provide a better experience across all devices. In other words, websites will look like shit LESS OFTEN if we just use the default font-size.

What exactly is an em, or a rem?

As best as I can tell, back when printers used movable type (so, we're back in the year 1440?), the width of a capital M was an "em", and that's where the term comes from, I guess? If we're going to reference history, I would have preferred a "g" (for Gutenberg), but I digress.

In web design, an "em" is a multiplier. So for example, if our inherited style has a 15px font-size and we specify 2em, then our font-size will become 30px. It's a percentage expressed without the %-sign, where 1em = 100%, 2em = 200%, etc. It's really no more complicated than that, although older browsers did wonky things to em's and for a while there were designers advocating to only use "%".

The "rem", or "root em", is just an "em" without any inherited multiplier, it only multiplies against the "root" (or default) font-size. So "rem" is technically just a measurement (like "px") and not a multiplier.

If we're using "rem" with the 62.5% trick, we're basically just using "px". I guess people feel more like artists typing "1.4rem" rather than "14px". Apparently, pixels are for nerds, artists use "em"? Until rem is more widely supported, designers are stuck trying to figure out the multiplier approach with "em", they have no "em" based measurement approach and will end up using "px".

Why not just use px?

This is already happening, and it's a perfectly valid approach. Modern browsers and smartphones provide competent zoom and font scaling. And many designers have found that "em" is too confusing to work with (as complicated as exclusively using "%") so they're just doing everything in "px".

That said, let's not throw out the baby with the bath water. The 62.5% trick is, well, stupid -- but specifying font-size as a multiplier of the default can be very useful.


KISS approach to font-size

1. Use the default font-size. Don't type "font-size" in a stylesheet unless you really know what you're doing. Trust that every device has provided sensible default font sizes. Readable text, like in this paragraph, should use the default font-size. Do NOT be presumptious and assume that you know better for all devices and all users.

2. Use "%" sparingly, and always with respect to the default font-size. For example, sometimes we need a menu or tool-tip that isn't as large as the normal text and there's no html to get the job done (like <sub> or <sup>). In these cases, use "%" sparingly in order to ensure that we will be respecting all devices and all users. Your job isn't to make people with glasses squint, it's to make sure that accessibly-sized fonts don't break the layout.

Also, "%" is more compatible across browsers than "em". I'm sorry designers, but that's the truth. If you're really concerned about old versions of IE, then don't use "em".

3. Use "px" sparingly. In some cases it's easier and more reusable to use "px" rather than a multiplier font-size (like "em", or "%"). For example, in an image gallery with fixed-size previews, it may be necessary to use "px" for the caption.

Keep the edge-cases on the edge

Basically, a good stylesheet should never mention font-size. And if it does, it should do so sparingly with "%" or "px". There's no use case for "em" other than making a website look like shit more often than had it not used "em".

That said, there exists edge cases, like if we need our font-size to scale with the viewport width. Basically, if we want a block of text to act like an image. In this case, we could use "vw" (viewport width).

<p style="font-size:2.5vw">This is relative to the viewport width!</p>
